## 项目一：直播电商微服务系统 (Mallive)

### 项目简介
基于 Go 语言开发的**直播电商微服务系统**，采用 DDD（领域驱动设计）+ CQRS 架构，实现了订单、库存、支付、履约等核心业务功能，支持高并发场景下的直播带货场景。

### 技术栈
- **语言**: Go 1.25
- **框架**: Gin (HTTP), gRPC, RabbitMQ
- **中间件**: MongoDB, MySQL, Redis, Consul, RabbitMQ
- **可观测性**: OpenTelemetry + Jaeger链路追踪
- **部署**: Docker, Kubernetes

### 核心职责与亮点

#### 1. 架构设计与实现
- 采用 **DDD 分层架构**：Adapters → Command/Query Handler → Domain → Ports
- 实现 **CQRS 模式**：分离读写操作，订单服务将 Command 和 Query 分离处理
- 使用 **Decorator 模式**：为 Command/Query 统一添加日志记录和性能指标收集

#### 2. 分布式事务与消息队列
- 基于 **RabbitMQ** 实现事件驱动架构
- 订单创建后自动发布 "订单已创建" 事件，支付服务消费该事件进行支付处理
- 实现 **RabbitMQ 链路追踪**：将 OpenTelemetry 的 TraceContext 注入到消息 Header 中，实现跨服务全链路追踪

#### 3. 服务注册与发现
- 集成 **Consul** 实现服务注册与发现
- 实现健康检查机制：每个服务定时向 Consul 发送心跳，维护服务实例状态

#### 4. 数据持久化
- 订单数据存储在 **MongoDB**，利用其灵活的结构存储订单商品列表
- 库存数据存储在 **MySQL**，保证数据一致性
- 集成 **Redis** 作为缓存层，优化热点数据查询

#### 5. 支付集成
- 集成 **Stripe API** 实现支付功能
- 创建支付链接后，通过 Webhook 接收支付回调，异步更新订单状态

#### 6. 可观测性建设
- 集成 **OpenTelemetry + Jaeger** 实现全链路追踪
- 统一日志收集：使用 Logrus 记录请求日志，包含 TraceID 方便问题排查
- gRPC 拦截器：自动在请求中注入和传递 TraceContext

### 关键技术点详解

#### 事件驱动流程
```
1. 用户创建订单
       ↓
2. Order 服务验证库存 (调用 Inventory gRPC)
       ↓
3. 创建 Pending 订单，存入 MongoDB
       ↓
4. 发布 "order.created" 事件到 RabbitMQ
       ↓
5. Payment 服务消费事件，生成 Stripe 支付链接
       ↓
6. 用户完成支付，Stripe Webhook 回调
       ↓
7. 更新订单状态为 "paid"，发布 "order.paid" 事件
       ↓
8. Kitchen 服务消费事件，开始履约
```

#### 全链路追踪实现
- gRPC 拦截器：在每次请求时自动注入 TraceID
- RabbitMQ Middleware：将 TraceContext 注入消息 Header
- HTTP 中间件：在 Gin 框架中提取和传递追踪信息

## 面试常见问题及回答思路

### Q1: 介绍一下这个项目的架构？
> 我们采用了微服务架构，分离了订单、库存、支付、履约四个服务。服务间通信使用 gRPC 同步调用和 RabbitMQ 异步消息两种方式。整体采用 DDD 的分层架构，Adapters 层处理外部请求，Command/Query Handler 处理业务逻辑，Domain 层包含核心领域模型。

### Q2: 为什么选用 RabbitMQ 而不是 Kafka？
> 一方面是因为项目规模适中，RabbitMQ 足够满足需求；另一方面是 RabbitMQ 支持复杂的路由规则和消息确认机制，适合我们这种需要可靠消息传递的场景。另外 gRPC + RabbitMQ 的组合在 Go 生态中生态更完善。

### Q3: 如何保证分布式事务一致性？
> 我们采用了最终一致性的方案。通过 RabbitMQ 的事件驱动来实现：订单创建后先写入本地数据库，然后发布事件；下游服务消费事件后执行相应操作。如果下游失败，可以配合重试机制和人工补偿来保证最终一致性。对于库存扣减这种强一致性场景，我们采用了分布式锁来保证。

### Q4: 如何实现全链路追踪？
> 我们在 gRPC 拦截器和 HTTP 中间件中自动注入 TraceID，并在请求上下文中传递。对于 RabbitMQ，我们自定义了 Middleware，将 OpenTelemetry 的 TraceContext 注入到消息的 Header 中。这样无论请求经过多少个服务，都能通过 TraceID 串联起来。

### Q5: 项目中遇到过什么技术挑战？
> 一个是 gRPC 在 K8s 环境下的服务发现问题。我们需要让服务正确注册到 Consul，同时要能获取到正确的 Pod IP。后来我们通过 K8s Service 来代替 Consul，实现了更简单的服务发现。
> 另一个是 RabbitMQ 消息积压导致的性能问题。我们通过优化消费者的并发数、增加死信队列处理失败消息来解决。
